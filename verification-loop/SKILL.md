# Verification Loop（実装後の検証ループ）

実装を「完了」と報告する前に、以下のループを**全ステップが通るまで**繰り返す。
途中で失敗したら修正し、ステップ1からやり直す（修正が新たな問題を生むことがあるため）。

## When to Use

- 機能実装・バグ修正・リファクタリングが「終わった」と思ったとき
- PR作成・コミット・ユーザーへの完了報告の直前
- 大きな変更の途中のチェックポイントとして

## How It Works

### Step 1: ビルド / 型チェック

プロジェクトのビルド・型チェックを実行する（例: `npm run build`, `tsc --noEmit`,
`cargo check`）。コマンドは package.json / Makefile / CI設定から特定する。

### Step 2: Lint / フォーマット

プロジェクトのlintを実行し、自分が変更したファイルに関する警告をゼロにする。
既存の無関係な警告は直さない（スコープクリープ防止）。

### Step 3: テスト

1. まず変更箇所に最も近いテストを実行（速いフィードバック）
2. 通ったらテストスイート全体（またはCIで走る範囲）を実行
3. 新しい挙動にはテストを追加。**失敗するテストをスキップ・削除して通すのは禁止**

### Step 4: 差分レビュー

`git diff` を読み直し、以下を確認する:

- デバッグ用の console.log / print / コメントアウトの消し忘れ
- 意図しないファイルの変更・フォーマットだけの差分
- ハードコードされた値・シークレット
- TODO のまま残した箇所

### Step 5: 実挙動の確認（可能なら）

テストだけで確信が持てない変更（UI・CLI・設定変更）は、実際に起動して
変更した挙動を最低1回は目で確認する。

## 完了報告のルール

- 全ステップの結果を**正直に**報告する（「テスト12件パス、lint警告0」のように具体的に）
- 実行できなかった検証があれば、その旨と理由を明記する
- 失敗が残っている状態で「完了」と言わない

## Examples

```
実装完了の報告例:
「実装完了です。検証結果: tsc型チェック通過、eslint警告0、
 ユニットテスト48件全パス（新規3件追加）、git diffに不要な変更なし。」
```

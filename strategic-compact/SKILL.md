# Strategic Compact（戦略的コンテキスト管理）

コンテキストの圧縮（/compact）は**タイミングと事前準備**で価値が決まる。
自動圧縮に任せて作業の途中で文脈を失うのではなく、区切りの良い場所で意図的に行う。

## When to Use

- suggest-compact フックや残量警告が出たとき
- タスクの一区切り（実装完了→レビュー開始など、フェーズの境界）
- これから大量のファイル読み込み・長い探索を始める直前

## How It Works

### 1. 圧縮に適したタイミングを選ぶ

**良いタイミング（フェーズ境界）:**

- タスクが1つ完了し、検証も終わった直後
- 調査・計画が終わり、実装に移る直前
- PR作成・コミット直後

**避けるべきタイミング:**

- デバッグの真っ最中（仮説・試行錯誤の履歴が消えると振り出しに戻る）
- 複数ファイルにまたがる編集の途中
- エラーの原因を特定した直後でまだ修正していないとき

### 2. 圧縮前に状態を外部化する

コンテキストは消えるが、ファイルは残る。圧縮前に以下を書き出す:

- 進行中タスクの状態 → 計画ファイルや TODO、`.claude/learnings.md`
- 重要な発見・決定事項 → メモリディレクトリまたはプロジェクトのドキュメント
- 「次にやること」を1〜3行で明文化

### 3. 指示付きで圧縮する

`/compact` は引数で「何を残すか」を指示できる。例:

```
/compact 現在のタスク（auth機能のリファクタリング）の進捗と、
settings.jsonの構造に関する発見は必ず保持して
```

### 4. 予防的なコンテキスト節約

圧縮以前に、消費を減らす習慣:

- 大きなファイルは必要な範囲だけ読む（offset/limit）
- 広い探索はサブエージェント（Explore）に委譲し、結論だけ受け取る
- 同じファイルを何度も読み直さない

## Examples

```
悪い例: コンテキスト残量5%の警告後、自動圧縮でデバッグ履歴が消え、
        同じ仮説を再検証して30分浪費。

良い例: ツール呼び出し50回の提案を受け、現在のバグ修正を検証完了まで
        進めてから learnings.md に原因と対処を記録し、/compact を実行。
```
